## What This Is

## The Protocol

### Phase 1: LIST ‚Äî Complete Enumeration

List all options literally. No filtering. No judgment. No ‚Äúthat‚Äôs unrealistic‚Äù yet.

The only rule: completeness. If it‚Äôs been proposed, considered, or even mentioned, it goes on the list.

**Output:** Exhaustive option set, unsorted, unfiltered.

### Phase 2: FORGE ‚Äî Iterative Constraint Application

Apply one constraint at a time. Always the most obvious constraint first ‚Äî the one that would occur to any reasonable observer.

For each constraint:

1. Apply the constraint cleanly
1. Verify: Did I lose anything essential? (Y/N)
1. If YES ‚Üí Reconsider constraint or mark exception
1. If NO ‚Üí Commit the cut
1. Document: which constraint, what was removed, verification result

**Constraint Types (apply in this order):**

|Order|Constraint Class     |Example Questions                                          |
|-----|---------------------|-----------------------------------------------------------|
|1    |Factual impossibility|Does this violate physics, law, or known constraints?      |
|2    |Resource reality     |Do we have the time, money, people, or skills?             |
|3    |Causal logic         |Would this actually produce the stated goal?               |
|4    |Temporal boundary    |Can this be done within the required timeframe?            |
|5    |Authority scope      |Are we authorized to do this?                              |
|6    |Dependency check     |Does this require something that doesn‚Äôt exist yet?        |
|7    |Proportionality      |Is the effort proportional to the outcome?                 |
|8    |Second-order effects |What does this enable or prevent later?                    |
|9    |Reversibility        |Can we undo this if it‚Äôs wrong?                            |
|10   |Elegance             |Does keeping this make the whole system clearer or muddier?|

**Critical rule:** Apply only one constraint at a time. Verify after each cut. Never batch-remove.

### Phase 3: STOP ‚Äî Inevitability Criterion

Stop when the remaining options feel **inevitable**, not just acceptable.

**The Inevitability Test:**

Ask: ‚ÄúCould a reasonable person looking at these options see any other path?‚Äù

- If **yes** ‚Äî keep cutting.
- If **no** ‚Äî stop. You‚Äôre done.

**Warning sign:** If cutting feels emotional rather than logical ‚Äî if you‚Äôre inventing constraints to justify a preference ‚Äî stop immediately. You‚Äôve left the forge and entered advocacy.

### Phase 4: CONFIRM ‚Äî CLEAN Check

Run the final set through the CLEAN checklist:

|Check         |Question                                                    |
|--------------|------------------------------------------------------------|
|**C**omplete  |Does this set cover all essential functions?                |
|**L**ogical   |Are the remaining options internally consistent?            |
|**E**fficient |Is each option necessary? (Remove one ‚Äî does the set break?)|
|**A**ctionable|Can these options actually be executed?                     |
|**N**ecessary |Is each option load-bearing, not just nice-to-have?         |

**Output:** Final option set, cut log (what was removed and why), CLEAN verification.

## The Philosophy

Adding is easy. Removing is hard. Removing the right things is everything.

Most decision processes fail not because they miss options, but because they keep too many. Every kept option is a tax on attention, a source of confusion, a delay in execution.

The forge doesn‚Äôt tell you what to choose. It tells you what you can stop considering ‚Äî so you can focus on what actually matters.
